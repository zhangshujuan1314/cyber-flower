# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**赛博养花 (Cyber Bloom)** is an AI-driven WeChat mini-program where users grow virtual flowers. Each flower has a unique AI-generated genome, personality, and growth path influenced by care history and the Chinese solar terms (二十四节气).

The repository is a monorepo with two main packages:

- `mini-program/` — WeChat mini-program frontend (native + TypeScript + MobX)
- `server/` — NestJS backend (MongoDB + Redis + AI services)
- `docs/` — PRD, architecture, UI design, task plan, setup guide, and WeChat review checklist

There is no root `package.json`; each package is managed independently.

## Common Commands

### Backend (`server/`)

```bash
cd server
npm install
npm run start:dev       # dev server with hot reload on http://localhost:3000
npm run build           # production build to dist/
npm run test            # run Jest unit tests
npm run test:e2e        # run end-to-end tests (test/app.e2e-spec.ts)
npm run lint            # ESLint with --fix
```

- The server requires MongoDB. Use `docker-compose up -d` inside `server/` to start MongoDB and Redis together.
- Swagger docs are served at `http://localhost:3000/api/docs`.
- API base path is `/v1`.

### Frontend (`mini-program/`)

```bash
cd mini-program
npm install --legacy-peer-deps
npm run compile         # type-check with tsc
npm run watch           # type-check in watch mode
npm run lint            # ESLint on .ts files
```

- The mini-program must be opened with **微信开发者工具** (WeChat DevTools) using the `mini-program/` directory.
- Before previewing, set your AppID in `mini-program/project.config.json` (`appid`).
- For local development with a custom backend, update `mini-program/services/api.ts` (`BASE_URL`).

### Docker

```bash
cd server
docker-compose up -d    # starts api, mongo, redis
docker build -t cyber-bloom-api:latest .  # production image
```

## High-Level Architecture

### Frontend (`mini-program/`)

- **Pages and routing** are declared in `app.json`.
  - Main pages live under `pages/` (garden, breed).
  - Secondary pages are in a sub-package under `sub-pkg/` (chat, collection, social, friend-garden, memorial, profile, privacy).
  - `preloadRule` preloads the sub-package when the garden page loads.
- **State management** uses `mobx-miniprogram` stores bound to pages via `mobx-miniprogram-bindings`:
  - `stores/gardenStore.ts` — flowers, seeds, current flower, season/solar term, care animations.
  - `stores/userStore.ts` — login state, user profile, resources (water/fertilizer).
- **Network layer** is centralized in `services/api.ts`, which attaches JWT tokens, handles 401 refresh, and provides `get/post/patch/delete` helpers.
- **Flower rendering** is implemented in `utils/flower-renderer.ts` and consumed by `components/flower-viewer/flower-viewer.ts`.
  - Uses a custom Canvas 2D renderer with layered sprites (shadow → pot → stem → leaves → calyx → petals → center).
  - Currently renders colored placeholders; real images are loaded via `currentImage` URLs generated by the backend.
  - Supports breathing animation, touch rotation, and care particle effects.
- **Season/节气 logic** is duplicated on client and server:
  - Client: `utils/season-utils.ts` provides `getCurrentSolarTerm()` and `getSeasonProgress()`.
  - Server: `core/season/season.service.ts` returns a simplified set of solar terms.

### Backend (`server/`)

- **Framework**: NestJS, structured as a modular monolith.
- **Entry point**: `src/main.ts` sets global validation pipe, CORS, `/v1` prefix, and Swagger at `/api/docs`.
- **Module layout** (`src/app.module.ts`):
  - `core/` — business domains: `auth`, `user`, `flower`, `breed`, `chat`, `social`, `season`, `collection`.
  - `ai-layer/` — global AI services: `LlmService`, `ImageGenService`, `ContentSafetyService`, `EmotionEngine`, `MemoryService`, `GreetingService`.
  - `shared/` — guards, filters, decorators, DTOs, types, and `cos.service.ts` for Tencent COS.
- **Database**: MongoDB via Mongoose.
  - Schemas are defined next to their modules (e.g., `core/flower/flower.schema.ts`, `core/flower/care-log.schema.ts`, `core/breed/seed.schema.ts`).
  - Shared TypeScript models live in `src/shared/types/models.ts` and are mirrored loosely in the mini-program stores.
- **Authentication**: WeChat `wx.login()` obtains a `code`, which the backend exchanges for `openid` via `auth.service.ts` and returns JWT access/refresh tokens. `JwtAuthGuard` protects most routes.
- **Growth engine**: `core/flower/growth.engine.ts` computes health/happiness/growth deltas from care actions, season preference, and emotion state, and decides when a flower advances to the next growth stage.
- **AI integrations**:
  - `llm.service.ts` calls the configured LLM endpoint (defaults to Anthropic Claude). Environment variables: `AI_LLM_ENDPOINT`, `AI_LLM_API_KEY`, `AI_LLM_MODEL`.
  - `image-gen.service.ts` generates flower stage images (currently a stub/placeholder path; intended for Stable Diffusion/FLUX).
  - `content-safety.service.ts` filters user input before seed generation.
  - `prompt-templates.ts` centralizes all LLM prompts (seed generation, chat, growth descriptions, image prompts, safety checks).
- **Rate limiting**: `ThrottlerModule` applies a global guard (100 requests per 60 seconds by default).

### Data Model Essentials

The core entities are `User`, `Seed`, `Flower`, `Genome`, `CareLog`, and `ChatMessage`.

- A `Seed` carries a `Genome` (species, colors, morphology, growth profile, rarity) and can originate from keywords, daily drops, gifts, hybrids, or events.
- A `Flower` is created when a seed is planted; it tracks `stage`, `health`, `happiness`, `stageTimestamps`, `personality`, and `visualState`.
- `CareLog` records every care action and its computed effect.
- Growth stages are: `seed → germinating → growing → budding → blooming → fruiting → dormant`.

## Configuration

Copy `server/.env.example` to `server/.env` and fill in:

- `WX_APPID` / `WX_SECRET` — required for WeChat login.
- `MONGODB_URI` — defaults to local MongoDB; MongoDB Atlas also works.
- `REDIS_URL` — required by some services, but many flows work without Redis running.
- `JWT_SECRET` — change in production.
- `AI_LLM_ENDPOINT` / `AI_LLM_API_KEY` / `AI_LLM_MODEL` — optional; if absent, seed generation and chat fall back to deterministic/mock responses.
- `AI_IMAGE_ENDPOINT` / `AI_IMAGE_API_KEY` — optional; image generation is not fully wired yet.
- `COS_SECRET_ID` / `COS_SECRET_KEY` / `COS_BUCKET` / `COS_REGION` — optional Tencent COS for image storage.

## CI

`.github/workflows/ci.yml` runs on pushes to `main`/`develop` and PRs to `main`:

1. `server-test` — `npm ci`, `npm run lint --if-present`, `npm test --if-present`, `npm run test:e2e --if-present`.
2. `server-build` — builds the server and uploads `server/dist` as an artifact.
3. `docker-build` — builds the server Docker image.

## Development Notes

- The project is in the design/MVP stage. Many AI features have scaffolding and fallbacks rather than full production implementations.
- The mini-program relies on WeChat DevTools for preview, compilation, and debugging; there is no browser-based dev server.
- When adding a new API endpoint, update both the server controller and the corresponding `mini-program/services/*.ts` file.
- Type definitions are intentionally duplicated between `server/src/shared/types/models.ts` and the mini-program stores/services to keep the WeChat build self-contained.

## 架构现状（真实 vs 文档）

对照 `docs/PRD.md`、`docs/tech-architecture.md`、`docs/task-plan.md`、`docs/setup-guide.md` 与真实代码，当前实现与文档存在以下差异：

### 1. 后端架构：单体 NestJS，非文档描述的微服务

- **文档描述**：`tech-architecture.md` 画出 `api-gateway/` + `services/user-service/`、`services/flower-service/`、`services/breed-service/`、`services/chat-service/`、`services/social-service/` 等多服务拆分架构。
- **真实代码**：`server/src/app.module.ts` 将 `core/auth`、`core/user`、`core/flower`、`core/breed`、`core/chat`、`core/social`、`core/season`、`core/collection` 作为 NestJS 模块直接注册，是典型的**单体模块化 NestJS 应用**，没有独立服务进程、没有 API Gateway 模块。

### 2. 生长引擎：纯规则引擎，非 AI 模型驱动

- **文档描述**：`tech-architecture.md` 中的 `GrowthEngine` 示例显示“AI修正：调用AI模型微调生长方向”，并提到“规则引擎 + AI模型混合架构”；`task-plan.md` 也将“AI生长引擎升级”列为 Phase 3 任务。
- **真实代码**：`server/src/core/flower/growth.engine.ts` 的 `calculate()` 完全基于数学规则：
  - 计算近 24 小时浇水/施肥次数；
  - 按季节偏好计算 `seasonModifier`；
  - 按情绪状态计算 `emotionModifier`；
  - 按正弦曲线计算生长进度；
  - 没有任何 LLM 或机器学习模型调用。

### 3. 领域类型：前后端不共享，各自重复定义

- **后端**：共享类型定义在 `server/src/shared/types/models.ts`（`Genome`、`Flower`、`Seed`、`CareLog`、`ChatMessage` 等）。
- **前端**：没有引用后端类型，而是在各文件中自行定义局部接口：
  - `mini-program/stores/gardenStore.ts` 定义 `Flower`、`Seed`、`SeasonInfo`。
  - `mini-program/stores/userStore.ts` 定义 `UserStats`、`UserResources`。
  - `mini-program/services/flower.ts` 定义 `FlowerData`、`CareResult`。
  - `mini-program/services/breed.ts` 定义 `SeedData`。
- 两端类型结构相似但不同步，修改后端模型后需要手动同步前端接口。

### 4. 文档描述与代码实现不一致的其他要点

| # | 文档描述 | 真实代码 | 说明 |
|---|----------|----------|------|
| 1 | 后端目录为 `server/services/*-service/` | 实际为 `server/src/core/*/` | 目录层级完全不同，不存在 `services/` 目录。 |
| 2 | 使用 RabbitMQ/腾讯云 CMQ 消息队列、Elasticsearch | 代码中无相关模块 | `app.module.ts` 只导入 Mongoose、Throttler 和各业务模块。 |
| 3 | 聊天人格基于花朵 DNA 与当前状态动态生成 | `chat.service.ts` 中 `CHAT_SYSTEM` 的 `{flowerName}`、`{health}`、`{happiness}`、`{growthStage}` 等占位符被硬编码为固定值（TODO 注释） | 当前回复未能真正体现具体花朵状态。 |
| 4 | 腾讯云 COS 图片上传/CDN 已集成 | `shared/cos/cos.service.ts` 是占位实现 | 仅返回拼接的 COS URL，未调用真实 COS SDK，相关 SDK 代码被注释。 |
| 5 | 图鉴系统展示已收集/未收集品种 | `collection.controller.ts` 返回硬编码 `{ flowers: [], collected: 0, total: 48 }` | 图鉴后端尚未实现。 |
| 6 | 聊天/图鉴/社交/个人中心为主包页面 | `app.json` 中这些页面放在 `subPackages`（`sub-pkg/`）里，仅花园和育种在主包 | 与 PRD 的页面层级结构有出入。 |
| 7 | 前端存在 `workers/` 目录用于图片预加载 | `mini-program/workers/` 目录不存在 | 文档中的 workers 设计未落地。 |
| 8 | 节气服务“精确到秒” | `season.service.ts` 使用简化的月份映射，只返回 8 个代表性节气 | 精确天文算法尚未实现。 |

> 总体而言，项目目前处于 **MVP/脚手架阶段**：文档规划了完整的 AI 驱动、微服务、云原生架构，但代码实现了单体 NestJS + 规则引擎 + 大量 TODO/占位逻辑，核心闭环（登录、育种、种植、照料、聊天降级）已跑通，生产级 AI 与基础设施尚未完成。
